4gl programs documenter

Introduction

During the life cycle of a project, the documentation that associates to the implementation (in 4gl code) to functionality it becomes rapidly unsynchronized.

This fact occurs due to form and place where this normally is stored.

The designer usualy to construct a document where it arrives for times at the detail to define the functions, its return values and parameters.

Is common to use CASE tools as the Power Designer putting inside of the processes definition the documentation descriptions of the program to be built.

In the initial phase of project development (if the manager have some carefull), it is probable that the documentation reflects the programmed code.

When dumb for a phase of the process most advanced, normally immediately after the first usable version, the chaos starts. The programmer has to decide and the tracker does not have time to control everything. Thus the source code gets modified, but as the programmer for definition does not like to modify documents this is immediately unsynchronized.

The more time passes more unsynchronized it is. Until it does not serve for nothing.

It would be easy to blame the programmers, however the documentation normally is hidden underneath of some server, in a obscure directory inside a document (normally in .doc format ) normally with sets of ten of pages. The context change is such that of fact me does not seem easy of it to demand similar effort.

The aproaches initiated with some work, as for example literate programming are in this market each time more used.

The Java is paradigmathic: The reference documentation of all API(s) is gotten by automatic processes with information contained in sources.

With this form, we easily note the existence of clear improvements, as for example:
  • Unique  tool used for programming and reference documentating (text editor, usually vi or emacs).
  • Easy visualization of documentation errors.
  • Increased productivity.
  • More communication inside of programming teams (given that all the functions will be registered, being this easily available).

This tool gives 4gl what already it exists in java it (with javadoc) or in the C/C++ (with projects as doxygen), or either:
  • The possibility of the reference documentation to be incluided in the source 4gl modules.
  • Generation of reference documentation in readable format.
  • Possibility of search.
  • Reference documents with diverse organizations.
Obviously, however the programmers and project managers have to create some discipline to allow that during the modifications to the programs, the documents are modified to. Inclusively he must yourself define some  patterns to the commentaries to insert.

Execution

One presents only tipical examples of execution with non graphical User Interface, called Command line User Interface (CUI).

To import information of sources

$ fgldoc -- ui=CUI -- action=ImportFgl -- import_directory=/tmp -- load_recursive

To import information of CSV(s)

$ fgldoc -- ui=CUI -- action=importCSV -- sheet_file=/tmp/comments.csv -- load_recursive -- it normalizes

To export to 4gl sources

$ fgldoc -- ui=CUI -- action=export2Fgl -- sheet_file = < fileName > -- export_directory = < directoryName >

To export to CSV

$ fgldoc -- ui=CUI -- action=export2Csv -- export_directory = < directoryName > -- export_source = < fglSourceName.4gl >

To export to HTML

$ fgldoc -- ui=CUI -- action=export2Html -- export_html_directory = < directory_name >

Components

fgldoc - Interface for execution of documentation shares

Component that serves for the execution of the actions provided by  4gl documenter. The majority of the documentation shares is executed by intermediary of this program.

It is build in Perl and the accpets the following execution options:

  • -- help: It defines the allowed options
  • -- ui={GUI | CUI }: User interface with which the application will be executed that could be:
    • GUI: Graphical User Interface (Mode Perl / Tk)
    • CUI: Command line User Interface. It does not have interaction with user. With this interface all options should be suplied in command line.
  • -- action = < ActionName >: Action to execute, that they could be:
    • importCSV: It imports the commentaries of file with a separating fixture generated by a spreadsheet for example.
    • importFgl: Imports the commentaries of 4gl source file.
    • export2Fgl: 4gl. Exports the commentaries to 4gl sources. The folowing parameters are needed -- exportDirectory and -- exportSource .
    • export2Csv: It exports the commentaries to a file with separating character.
    • export2Html: It exports the commentaries to a set of html file duly linked
  • -- export_directory = < dirName >: In the case of exportation of the commentaries for sources of 4gl it must contain the name of the directory where to generate the files.
  • -- exportSource = < fglname >: In the case of exportation of the commentaries for sources of 4gl it will go to contain the name of the program source.
  • -- quiet: It removes all and any message sent for standard output.
  • -- insertEmptyComment: If assigned it indicates that it must be inserted an empty commentary in a function that has not defined it in the repository. Used in the option of exportation for sources 4gl.
  • -- sheet_file = < SCVfilecName >: It indicates the name of the file with CSV separator to use in the case of the importation and exportation for files with caracter separating.
  • -- it normalizes: It indicates that the loaded information of the CSV file must be normalized and not only kept in the table with equal structure of the file.
  • -- clean_repository: If assigned it indicates that the repository must be clean before loaded the information.
  • -- send_line_to_log
  • -- package_name=s: It indicates the name of package to insert during of 4gl sources parsing.
  • -- load_recursive: In the case of the shipment of information of 4gl(s) it indicates that it must be loaded the files of the directory and all the corresponding sub-directories.
  • -- import_directory=s: It defines the directory where they are 4gl(s) of where if it intends to load the information.
  • -- export_html_directory = < htmlDirectoryName >: The directory indicates which to where the information in format HTML will be exported / generated
  • -- to ignore_exist_dir: If assigned it indicates that the program must ignore if the directory already exists. Used for the exportation for HTML

Cui  user interface

CUI is an acronym that means Command line User Interface.

The goal is to make possible the batch execution of the allowed operations.

Acceded when the program executed  with option --ui=CUI.

A time that after its execution will not have to exist any interaction with the users, all the necessary options must be supplied by command string or to be available in the parameters file (normally .fgldocrc).

Import -> Csv

It imports commentaries of a filing-cabinet CSV (Character Separated) for the repository in relationary model (database).

The importation is made in two phases:
  • Load of the file for one database table with the same structure.
  • Normalization of the information for the defined relational model.

It allows to define:
  • File  to load:   Full name of the file to load giving access to a File Chooser for the choice. For default its extension will be txt.
  • Cleanness of the repository: Not used
  • Normalization: When unactivated 2ª phase is not executed.
  • To send string for log: When activated, each loaded string is sent for log.

If the user to load in the Cancel < button > or to close the window, the operation is not executed coming back to the main menu.


The file must have as separator the < TAB > character between columns and < Enter > between strings.

It must contain 7 columns whose information must be (for order):

  • moduleName: Name of the 4gl module.
  • idProcess: Process code just like in p4gl_process table
  • functionName: Name of the function
  • parametro: Parameter
  • return: Description of the return values.
  • call  :
  • comments: Text comment to the function

Import -> 4gl Sources

It imports the available information and commentaries in 4gl source.

The information to configurate in the window is:
  • Name of package: Name of package the one that the 4gl modules will be associated. For now is used just for diferentiate modules.
  • Location of the directory: Full name of the directory below of which 4gl sources to parse are available.
  • Stack of modules: Full stack of the 4gl modules of that it is intended to parse.
  • Load comments: It indicates if it intends or not to load in the repository the fgldoc commentaries.
  • Parse Only: It only serves to test if parsing or is done successfully not loading any information in the database.
  • Repository: Information necessary to establish the connection with the database.
  • Shipment of table use: If activated any information on table usage is loaded in the repository.
  • Load parameters: If activated it indicates that parser it must load the parameters of the functions in the repository.

Exports all the information available in the repository to a text file with character separator.

Export -> 4gl Sources

Inserts the information available in the repository in the 4gl sources as fgldoc comments.

The module comment is inserted at the begining of the file.

The comments for each function is inserted just before its implementation.

It allows to configure the following options:
  • Directory : The place (directory name) where the 4gl source to insert the comments is located.
  • 4gl module  : Presetents all the 4gl(s) that are in the chosen directory so the user can chose the one where to insert the comments.
  • fgldoc comment replacemenet : If activated it substitutes eventual fgldoc that exist in the file, otherwise  leaves them and creates a new one.

If the user press  the <Cancel>  button or close the window, the operation is not executed and return to the main menu.

Its is the executable program that does the file parsing (with lexical and syntathic analisys) of the 4gl source program and all the special coments embedded.

The name of the executable file is p4gl.

Allows the following execution options:

  •    - s: It indicates that it is intended to load informs ç to it ã contained in coment á normal rivers (n ã eats it ç ados with {**).

This executable is, under the point of view fgldocumenter tool, the most important part. When a project  will be well organized and structured, the functional documentation wil being made by the programmers in each source program and loaded to the repository using this tool.

The parser is implemented with the reserved word limitation such as great part of 4gl(s) compilers existing (except of Informix and 4js).

This way, the sources  loaded informs should not contain reserved words as name of identifiers (example: you cant using display as variable name).

If such to occur and you even intend to load the information of these programs, it can be used one special  comment  that, when inserted makes that  that parser ignores the 4gl code inserted between it. This  special comment is  {IGNORE } and ignores the code until it finds a new  equal comment.

When if to try to load the information  with these reserved words without without special  IGNORE comment, the parser alert to the existence of a  syntax error..

Documentation refactoring cycle

The goal of development of this tool was the need of knowing "something" about 4gl projects to make some refactoring in that code.

So, we gain some experience with this kind of processes-

Thus, given the caracteristicas of the tool we can say that this allows us to document a 4gl projecto in an iterative and evolutive form.

Such happens beacause its possible to do imports and exports in both ways with severall information formats allowing to modify it in this formats and loaded again  (CSV, RDBMS, 4gl) being able to be converted for another format (with the exception of the generated HTML that is unidireccional).

Thus, we think that to document a 4gl project we could follow the steps:
  • Load a first version of 4gl information with origin in 4gl modules using the import from 4gl sources to the repository: After this action all the modules, functions and parameters should be kept in the repository.
  • Generate a CSV file with this information loaded.
  • Edit the information in Excel or other spreadsheet.
  • Complete the repository information with the one edited in the spreadsheet.
  • Export the information to 4gl source modules.
  • Modify the coments in the 4gl.
  • Load the information from 4gl sources when needed.

New project documentation

If we are starting with a new 4gl project, the team should be duly formed to insert the fgldoc comments during the program work.

Thus, is prevented unnecessary refactorings and the easiness of maintenance is maginified.

If used the stack of todo tasks for carrying through, can be an important aid in the stream / program process, given that the programmer can  register in the sources that he is  writing (without necessity change to other environment) which the tasks that is leaving for later implementing (therefore during the development our mind are gone remembering details that we can not implement at that time because we are implementing another functionality).

Principios

The applications normally have three types of documents: structural and manual specifications, documents for the users.

The scope of this document is arrested only with the documents that describe the applications structurally.

Documentation created in autonomous documents normally is desactualizados and modified only when the management defines it as with priority task (what its fôr state only happens when such that already does not have any utility).

Of this form we think that the best form will be the documentation to be the position of the programmers and preference next to the code that if is to register (or either in source of 4gl).

Therefore, the example of javadoc we decide to perfectly create a definite tool that on the basis of standards very defined well extracts the information absorbed in the code and generates documents with format specifies and.

We must therefore try to follow the following principios:

  • The structural documentation is defined under the form of commentaries of documentation written in the code marked source and some documents as acessiveis specifications of these documents.

  • References the alterations efectuadas in determined version must be kept in the system of control of versions.

The commentaries do not have to be of the exclusive responsibility of none of these entities. This is partilhada between them consonant the phase where projecto if finds,

It is important to notice that to a large extent of the projectos, the programs source are inherited already construidos and many times not registered. Of any form, of certainty they are not registered with the nomenclature that we use.

Ideally the documentation must be written by the programmers to the measure who implement (or they modify) the functions. If to exist a methodology of initial writing of a skeleton being later each function implemented for the programmers, must be the responsible one for the construction of the skeleton to write it.

When this documentation is initiated already after the application being in functioning (in a process of refactoring or simply of documentation), must be efectuada under the form of taken over on a contract basis tentanto to the maximum that is most exhausting possivel.

After the inserted documentation, during the phase of maintenance of projecto, the programmers have to modify all the documentation in that he relates to the efectuadas modifications. In the case of resolution of bug(s) these must be registadas in the control of versions.

With this in mind, these principios try to describe the commentaries for documentation. They must be understood as suggestions instead of rigidos requirements to follow until the a exhaustion.

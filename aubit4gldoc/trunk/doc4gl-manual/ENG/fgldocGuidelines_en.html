Basic principles

The applications normally have three types of documents: 

  • structural specifications
  • manual specifications
  • documents for the users

The scope of this document is limited to documents that describe the applications structurally.

Documentation created in autonomous documents normally is obsolete and modified only when the management defines it as priority task (what usually only happens when we already don't have any resources available).

We think that the best form will be the documentation to be the position of the programmers and preference next to the code that if is to register (or either in source of 4gl).

Following the example of Javadoc, we decide to create a definite tool that on the basis of well defined standards extracts the information absorbed in the code and generates documents with format specified.

We must therefore try to follow this principles:

  • The structural documentation is defined in the form of documentation comments, written in the source code and some documents as acessiveis specifications of these documents.

  • References of the changes efectuadas in determined version must be kept in the versioning control system .

Who is the owner of the documentation comments - Designers, Documentation writers or Programmers ? The comments do not have to be of the exclusive responsibility of none of these entities. This is divided between them consonant the phase where project if finds.

It is important to notice that to a large extent of the projects, the programs source are inherited and already exist, and many times are not registered. In any case, they are probably not registered in the format that we use.

Ideally, the documentation must be written by the programmers who implement or modify the functions. If there is a methodology of initial writing of a skeleton, and later each function implemented for the programmers, then the responsible person for the construction of the skeleton should write it.

When documentation effort is initiated after the application is in use (in a process of redesign or to document it), must be efectuada under the form of taken over on a contract basis tentanto to the absolute maximum.

After the documentation is inserted, during the phase of maintenance of the project, the programmers have to modify all the documentation that relates to performed modifications. In the case of resolution of a bug, these must be registered in the version control.

With this in mind, these principles try to describe the comments for documentation. They must be understood as suggestions instead of rigid requirements to follow without question. There always exist alternative ways, that if prove to be more efficient, should be used. Please suggest them to person responsible for this document.

Terminology

Comments for documentation (doc comments)

Comments special written in the 4gl source code and delimited by { * *... } symbols. These comments are processed by the p4gl tool to create the documentation in a relational database. Developed using flex (Lex), bison (Yacc), C and Perl.

fgldoc

The tool that dynamically generates the documentation from the loaded data in the relational formated repository. It was developed in JSP and Java.

General form of a documentation comment

A documentation commentary is made from two parts: following description of zero or more tags, with a blank string between these sections: 

 { * * That is the part of the description of the
documentation comment

@process SUBSCRIBERS }
  • The first string is indented to be lined up with the code that follows the comment and starts with symbol of beginning-of-commentary ({**) followed by one return (ENTER).
  • The following strings start with an "*" or only with a space. When it exist, the "*" is removed and substituted with a space (" ") character.
  • A string of blank comment between the description and the list of tags must be inserted such as shown above.
  • To insert strings blank between related tablets of tags
  • The ending string starts with symbol of end-of-commentary (}) followed by one return character (ENTER).

Strings with more than 80 characters must be to be left. If there will be more than one paragraph in the documentation comment, separates the paragraphs with tag <P>.

Descriptions

First phrase

The first phrase of each documentation comment must be a summary, containing a concise but full description of the item that is to be described. The Javadoc tool uses this first phrase for the summary of the item. This becomes important for writing of initial information that if obtains to describe immediately.

This phrase finishes with the first full stop (".") followed by a space, tab, end of string or in first tag found. In the example that if follows the first phrase finishes in Prof.: 

 / * * * This is a simulation of experience of the Prof.
Sparrow.  * /

We can insert a full stop using meta-characters of the HTML to obtain this objective, as in following example: 

 / * * * This is a simulation of experience of the Prof.
&nbsp;  Sparrow.  * /

Style guide

Following are some suggestions and conventions for writing of the descriptions in the documentation comments.

  • Use <CODE> for words key and names

The private words must be surrounded by tags <CODE>... </ CODE> when included in a description. This includes:

  • 4gl language reserved word
  • Names of functions
  • Names of 0 variable (global, local and parameters)
  • Examples of code
  • Omit parentheses for the general form of the functions

When mentioning function itself in the comment of the function, one must always omit the parentheses and the respective parameters. It is preferred to use the word method if you intend to use the function name.

Example:

  • The function add allows to insert items. - Recommended
  • Function add(item) allows to insert items - Avoid

  • Use expressions instead of full conjuncts, in the interest of the brevity

This rule must primarily be applied when you define tags, as for example the @param

  • Use third person (descriptive), not a second person (prescriptive) form

The description should be in 3ª declarative person instead of 2ª imperative person.

Example:

  • It returns the number: Recommended
  • You return the number: Avoid

  • Descriptions of functions must start with verbal expression

A method implements an operation, thus normally starts with a phrase.

  • Gets the name of the customer: Recommended
  • This function gets the name of the customer: Avoid

  • Add descriptions for beyond the name the function

The best names of functions are automatically registered. However, if the commentary repeats the name of the function in a phrase, it does not add any information. The ideal comment goes beyond the these words and must always complement it with information that is not immediately (or at all) obvious from the name of the function.

Example:

Avoid - 

 { * * * Affects the name of the customer * } function
setClientName(name)

Recommended - 

 { * * * Stores the name of the customer in the modular 0
variable and later saves it in the table customers * } function
setCliemntName(name)
  • Avoid abbreviations

Use "also known as" instead of "aka", or "to be more specific" instead of "i.e".

Conventions for tagging

Order of tags

Tags must be included in the following order: 

 * @author * @version * * @see @param * @return * * @since *
@deprecated * @todo

It is preferable to divide tags for related tablets.

When same tag is inserted multiple times, it  must be separated from others to enhance readability.

@param tag must have the same order as corresponding function parameters.

Description of comment tags

@author

Omit if author is unknown. It can be used expansões efectuadas for the control of versoes. If several exist, it must be commanded for log order.

@version

It can be used to expand the version control. It corresponds to the described item in source code, and not to the version of the executable that results from compilation of several source modules.

@param

This tag must be followed by the name of the parameter (not it's type). Although it is redundant with the declaration of parameters, it is the only way to associate a comments with the parameter. For convention the first substantive of the description must reflect the type of information that it stores (it can be preceded by articles as " ").

Example: 

 * string of formatting of the number @param in one the number
to format * @param strUsing

Do not insert it between <CODE> </CODE> tags, because the scanning tool already creates them.

Preferably, use an expression in conjunction.

@return

In methods that do not return any value, it must be omitted.

When the method returns different values, several tags for each occurrence of RETURN must be created.

@deprecated

Indicates a method as not to be used, to indicate it will probably be removed in next version.

@since

It marks the version (when this is known ) from which the method was added.

Insert it only if this is known to be correct.

@todo

Defines a functionality to be implement in this method, in the future. It allows the programmer to mark task that must be implemented in the function, but that it is not opportune to implement at the moment, or only to note a problem or because it does not wants to break compatibility.

This way we can document what is lacking in implementation.
